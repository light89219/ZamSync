# ZamSync Engineering Guarantees

ZamSync is a **Causally-Ordered Replicated Log** infrastructure. It provides specific guarantees to the application layer to ensure eventual consistency across distributed nodes.

## 1. Core Guarantees

### G1: Causal Order Consistency
The engine guarantees that events originating from the same node are delivered and replayed in the exact order they were created.
- *Invariant*: If node A produces $E_{A,1}$ followed by $E_{A,2}$, any peer will apply $E_{A,1}$ before $E_{A,2}$.

### G2: Deterministic Total Ordering
Even if events from different nodes arrive in different orders via the network, the engine ensures a **Deterministic Global Order** during projection replay.
- *Mechanism*: Tie-breaking concurrent events using **Hybrid Logical Clocks (HLC)** and **Node IDs**.
- *Result*: Given the same set of events, all nodes produce the same state projection.

### G3: Atomicity and Durability
Every event is written to an append-only Write-Ahead Log (WAL) with CRC32 verification before being acknowledged or applied.
- *Mechanism*: `fsync` boundaries ensure that "Submitted" means "Persisted".

## 2. Convergence Model

ZamSync follows an **Eventual Consistency** model.
- **Truth**: The total set of all events generated by all nodes.
- **Knowledge**: Tracked via **Version Vectors**.
- **State Projection**: A deterministic function $S = f(Log, Order)$.

## 3. Conflict Strategy: Deterministic Log Sequencing

ZamSync does **not** resolve semantic conflicts (e.g., two people editing the same medical record). It resolves **sequencing conflicts**.
- The engine guarantees that all nodes see the same sequence of events.
- The application (StateStore) must be written to be a **deterministic reducer**.
- If two updates to the same field are concurrent, the reducer will receive them in a deterministic order (e.g., $E_{A}$ then $E_{B}$ on every node).

## 5. ZamSync Consistency Contract

This contract defines the strict relationship between the ZamSync Engine and the Application Layer (`StateStore`).

### Engine Guarantees (The Infrastructure)
1. **Total Deterministic Order**: The engine guarantees that for any two events $E_1$ and $E_2$, all nodes will agree on whether $E_1 < E_2$ or $E_2 < E_1$ based on `(HLC, NodeId)`.
2. **Monotonicity**: Local events are assigned HLCs that are strictly increasing.
3. **Reliable Prefix Replay**: If a node has replayed up to version vector $V$, it is guaranteed to have seen all causal ancestors of the events represented by $V$.

### Application Requirements (The Projector)
1. **Deterministic Reduction**: The `StateStore` must be a pure function $S_{n} = f(S_{n-1}, E_n)$. No external state, random numbers, or unsynchronized clocks can be used during replay.
2. **Idempotency**: The application must handle redelivery of the same event (identified by `(OriginNode, Seq)`) without side effects.
3. **Semantic Conflict Resolution**:
    - For simple fields, the application may use the **HLC Order** as a "Last-Writer-Wins" mechanism.
    - For complex data (e.g., medical logs), the application should use **Multi-Value** or **CRDT** patterns to avoid data loss during concurrent updates.

## 6. Definitions of "Correct State"

A node's state is **Correct** if it is the result of replaying the deterministic sequence of all events in its local knowledge.
A system is **Converged** when all nodes have the same Version Vector and have replayed their logs.
